What is Huffman Coding?

Before we get started, let's quickly discuss what exactly Huffman coding is. When we transmit information, we generally need to convert some sort of data (text, pictures, etc.) into binary. To do this, we assign codes to help us distinguish between different pieces of data.

For example, if we had the string "abca", we might assign codes like: $a$ = 00, $b$ = 01, $c$ = 10. This would make it so that our binary encoding of "abca" is "00 01 10 00".

But what if we wanted to encode "aabaacaa"? If we used our original encoding where we use two bits for each character, we would be treating $a$ and $c$ with equal importance, even though $a$ appears much more.

Wouldn't it be more efficient if we used fewer bits for $a$ and more bits for $c$? This is where Huffman coding comes in.

Huffman coding is a lossless data compression algorithm that assigns variable-length codes based on the frequencies of our input characters.

In order to determine what code to assign to each character, we will first build a binary tree that will organize our characters based on frequency.

Time Complexity Of Huffman Coding

Huffman coding achieves efficient compression by using a min heap to repeatedly select the two characters with the lowest frequencies while constructing the encoding tree.

The frequency table is first converted into a priority queue where each insertion operation takes O(log n) time.

During tree construction, the algorithm repeatedly removes the two minimum-frequency nodes from the heap and merges them into a new internal node.

Since each extraction and insertion operation requires O(log n) time, building the complete Huffman tree takes O(n log n) complexity overall.

The space complexity of Huffman coding is O(n) because the algorithm stores character frequencies, heap nodes, and the final binary tree structure in memory.

Additional space is also required for storing generated binary codes used during compression and reconstruction of the text data.

Building a Huffman Tree

As an example, let's encode the string "basketball". Before we can start encoding, we will build our Huffman tree for this string, which will in turn show us what binary encoding we will use for each character.

To start, we need to count the frequency for each character in our string and store these frequencies in a table.

We will use this table to add nodes and edges that will build up our tree.

First, we start by adding leaf nodes for two of the characters that occur the least. In this case, we have a tie between $s$, $k$, $e$, and $t$, so we'll just arbitrarily choose $s$ and $k$.

When we add in our leaves for $s$ and $k$, we will attach them to a parent node for a new pseudo-character "sk". This pseudo-character represents occurrences of $s$ or $k$ so its frequency is equal to the frequency of $s$ plus the frequency of $k$.

We will also update our table to include our new pseudo-character. We can get this by simply merging the columns of $s$ and $k$.

With a now reduced table, we can repeat this process again for our updated values.

We still have two singletons left, $e$ and $t$, so those are the next two leaves we combine.

Next, we can combine the two repeated letters $b$ and $a$, each of which occurs twice.

As before, we'll attach those leaves to a parent node for the new pseudo-character "ba".

Finally, we'll update our table to reflect our new pseudo-character.

It looks like our tree is coming along, but it doesn't quite have everything yet.

To keep going, we can repeat this process again...

...and again

...and again

...until our table is only left with one value, a pseudo-character containing all of our original characters. This means we're done building our Huffman tree!

So how do we use this tree to assign codes?

Given our Huffman tree, to determine the binary code that we will use for any particular character, we can simply walk from the root to our character's leaf node, taking note of when we traverse left and when we traverse right.

As we walk from root to leaf, we will denote a left traversal with "0" and a right traversal with a "1".

For example, say we wanted to find the encoding for $s$, which only occurs once in "basketball".

In our walk from root to the $s$ leaf, we go left, right, and left. This means we will use three bits to encode $s$ as "010".

What about our encoding of a more frequently used character like $l$?

For our walk to $l$, we traverse left just once more and stop. This means we will only use two bits to encode $l$ as "00".

It seems like our tree works - hooray for efficiency!

If we do this for all of our characters, we get our full binary encoding scheme. Let's compare this new Huffman scheme against a naive encoding scheme where we just arbitrarily assign binary codes.


char $b$ $a$ $s$ $k$ $e$ $t$ $l$
freq 2 2 1 1 1 1 2
old code 000 001 010 011 100 101 110
new code 110 111 010 011 100 101 00

Using the naive scheme, encoding "basketball" would take 30 bits. Using our Huffman scheme, we only use 28 bits to encode, which is a modest but real improvement.

A small thing to note: as we were building our tree, when choosing our two least frequent characters in our table, we repeatedly had ties between three or more characters. When this happened, we would choose two of our tied elements arbitrarily.

By doing this, we can see that our arbitrary choice will change our tree. This means we can actually get multiple different trees from the same input string. For example, we could have initially chosen to start with $e$ and $t$ instead of $s$ and $k$. If we had done this, we would get a very similar tree but the low-frequency leaves would be grouped differently.

While these trees might differ in their arrangement and shape, they are all valid Huffman trees. Since the algorithm is based on frequencies, it does not really matter which one-bit ties get grouped first. The exact codes may move around, but the overall compression stays essentially the same.

This completes the static Huffman example for "basketball".
